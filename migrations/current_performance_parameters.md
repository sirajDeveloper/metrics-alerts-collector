## Критические проблемы для производительности

### 🔴 Критично (влияет на производительность запроса):

1. **`work_mem = 4MB`** - слишком мало для запросов с сортировками и window functions
   - **Рекомендация**: Увеличить до 64-128MB
   - **Влияние**: Запрос может использовать временные файлы на диске вместо памяти

2. **`shared_buffers = 128MB`** - слишком мало для кэширования данных
   - **Рекомендация**: Увеличить до 2GB (25% от RAM)
   - **Влияние**: Больше обращений к диску, меньше данных в памяти

3. **`random_page_cost = 4.0`** - слишком высоко для SSD
   - **Рекомендация**: Уменьшить до 1.1-1.5
   - **Влияние**: Планировщик может выбирать Seq Scan вместо Index Scan

4. **`effective_io_concurrency = 1`** - слишком мало для SSD
   - **Рекомендация**: Увеличить до 200
   - **Влияние**: Неэффективное использование параллельных I/O операций

### 🟡 Рекомендуется улучшить:

1. **`effective_cache_size = 4GB`** - можно увеличить до 6GB
2. **`default_statistics_target = 100`** - можно увеличить до 200 для более точных планов
3. **`maintenance_work_mem = 64MB`** - можно увеличить до 1GB

## Команды для оптимизации

```sql
-- Критически важные изменения
ALTER SYSTEM SET work_mem = '64MB';
ALTER SYSTEM SET shared_buffers = '2GB';
ALTER SYSTEM SET random_page_cost = 1.1;
ALTER SYSTEM SET effective_io_concurrency = 200;

-- Рекомендуемые изменения
ALTER SYSTEM SET effective_cache_size = '6GB';
ALTER SYSTEM SET default_statistics_target = 200;
ALTER SYSTEM SET maintenance_work_mem = '1GB';

-- После изменений требуется перезапуск PostgreSQL
-- (кроме параметров с context = 'user' или 'sighup')
```

## Примечания

- Параметры с `context = 'postmaster'` требуют перезапуска PostgreSQL
- Параметры с `context = 'sighup'` требуют `SELECT pg_reload_conf();`
- Параметры с `context = 'user'` можно изменить в сессии через `SET`
